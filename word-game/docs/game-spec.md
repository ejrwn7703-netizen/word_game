## 1. PRD — 제품 요구사항 정의서

### 1.1 프로젝트 개요

실시간 멀티플레이어 단어 맞추기 웹 게임.  
방을 만들어 친구들을 초대하고, 화면에 나오는 단어 힌트를 가장 먼저 맞추는 플레이어가 점수를 얻는 방식이다.  
추후 다양한 게임 모드를 추가할 수 있도록 확장 가능한 구조로 개발한다.

### 1.2 목표

- 별도의 앱 설치 없이 링크만으로 여러 명이 동시에 참여할 수 있는 게임을 제공한다.
- 회원가입 없이 닉네임만으로 즉시 플레이 가능하게 한다.
- 추후 다른 게임 모드로 확장 가능한 구조를 갖춘다.

### 1.3 타겟 사용자

- 친구·지인과 온라인으로 함께 게임을 즐기려는 소규모 그룹 (2~10명)
- 계정 생성 없이 빠르게 합류하기를 원하는 사용자

### 1.4 핵심 기능 요약

| 기능 | 설명 |
|------|------|
| 닉네임 입장 | 로그인 없이 닉네임만 입력해 게임 참여 |
| 방 생성/참여 | 방을 만들고 링크로 공유해 친구 초대 |
| 실시간 경쟁 | 같은 화면을 보며 먼저 맞추는 사람이 점수 획득 |
| 힌트 시스템 | 0초 글자 힌트, 15초 카테고리 힌트 |
| 다중 종료 조건 | 10점 달성 / 5분 경과 / 30문제 소진 중 먼저 충족 시 종료 |

### 1.5 범위 외 항목 (현재 버전에서 제외)

- 회원가입 / 로그인 / 마이페이지
- 게임 기록 저장 / 전적 통계
- 단어 카테고리 선택 필터
- 방장이 커스텀 단어를 직접 등록하는 기능
- 자유 채팅 기능 (정답 로그는 포함)
- 모바일 앱 (웹 브라우저만 지원)

## 2. SRS — 소프트웨어 요구사항 명세서

### 2.1 기능 요구사항

#### 2.1.1 닉네임 입력 및 입장

| ID | 요구사항 |
|----|---------|
| FR-01 | 사이트 최초 접속 시 닉네임 입력 화면을 표시한다. |
| FR-02 | 닉네임은 1~10자이며, 공백만으로 이루어질 수 없다. |
| FR-03 | 닉네임 입력 후 "방 만들기" 또는 "방 코드 입력 후 입장" 중 선택한다. |

#### 2.1.2 방 시스템

| ID | 요구사항 |
|----|---------|
| FR-10 | 방 만들기 시 6자리 영문·숫자 혼합 방 코드를 서버에서 중복 없이 생성한다. (예: ABC123) |
| FR-11 | 방 입장 링크 형식: `https://{도메인}/?room=ABC123` |
| FR-12 | 방 코드를 직접 입력하거나, 링크 클릭으로 닉네임 입력 후 자동 입장한다. |
| FR-13 | 대기실에서 현재 참여자 닉네임 목록을 실시간으로 표시한다. |
| FR-14 | 방장(최초 생성자)만 게임 시작 버튼을 활성화한다. |
| FR-15 | 참여자가 2명 미만이면 게임 시작 버튼을 비활성화하고 안내 문구를 표시한다. |
| FR-16 | 방장이 대기실에서 퇴장하면 입장 순서가 다음인 플레이어가 자동으로 방장이 된다. |
| FR-17 | 게임이 이미 진행 중인 방에는 신규 입장을 허용하지 않는다. |

#### 2.1.3 힌트 시스템

**힌트 타임라인 (문제당):**

| 시점 | 내용 | 조건 |
|------|------|------|
| 0초 | 글자 힌트 표시 | 항상 |
| 15초 | 카테고리 힌트 추가 표시 | 아직 아무도 정답을 맞추지 못한 경우 |
| 20초 | 타임아웃, 정답 공개 | 아무도 정답을 맞추지 못한 경우 |

**글자 힌트 생성 규칙:**

| 단어 길이 | 힌트 방식 | 예시 |
|-----------|-----------|------|
| 2글자 | 앞 1글자 또는 뒤 1글자 랜덤 공개, 나머지 1글자는 `ㅇ` | `세ㅇ` 또는 `ㅇ왕` |
| 3글자 이상 | 앞 2글자 또는 뒤 2글자 랜덤 공개, 나머지는 `ㅇ` | `세종ㅇㅇ` 또는 `ㅇㅇ대왕` |

| ID | 요구사항 |
|----|---------|
| FR-20 | 힌트 방향(앞/뒤)은 문제마다 서버에서 랜덤으로 결정한다. |
| FR-21 | 공개되지 않은 글자는 글자 수만큼 `ㅇ`으로 표시한다. |
| FR-22 | 힌트 아래에 전체 글자 수를 표시한다. (예: `4글자`) |
| FR-23 | 15초 카테고리 힌트는 정답이 이미 맞춰진 문제에는 표시하지 않는다. |

#### 2.1.4 게임 진행

| ID | 요구사항 |
|----|---------|
| FR-30 | 게임 시작 시 단어 목록에서 랜덤으로 섞어 출제 순서를 결정하며, 같은 게임 내에서 중복 출제하지 않는다. |
| FR-31 | 모든 플레이어는 동시에 같은 문제를 본다. |
| FR-32 | 정답 비교 시 앞뒤 공백을 제거(trim)한 후 비교한다. |
| FR-33 | 정답은 띄어쓰기 없이 입력하는 것을 기준으로 한다. (단어 목록도 동일 기준) |
| FR-34 | 정답을 맞춘 즉시 타이머를 멈추고, 모든 참여자에게 정답자 닉네임과 정답을 표시한다. |
| FR-35 | 정답 공개 후 3초 대기, 자동으로 다음 문제로 전환한다. |
| FR-36 | 20초 타임아웃 시 정답을 공개하고 3초 대기 후 다음 문제로 전환한다. |
| FR-37 | 오답 입력은 점수에 영향을 주지 않는다. (패널티 없음) |
| FR-38 | 문제 전환 전 3초 카운트다운("3초 후 다음 문제...")을 화면에 표시한다. |

#### 2.1.5 점수 시스템

| ID | 요구사항 |
|----|---------|
| FR-40 | 정답을 먼저 맞춘 플레이어에게 +1점을 부여한다. |
| FR-41 | 점수판은 게임 화면에 항상 표시하며, 점수 높은 순으로 정렬한다. |
| FR-42 | 점수는 모든 플레이어에게 실시간으로 동기화한다. |

#### 2.1.6 게임 종료

**종료 조건 — 셋 중 하나를 먼저 충족 시 게임 종료:**

| 조건 | 내용 |
|------|------|
| 점수 조건 | 특정 플레이어가 10점 달성 |
| 시간 조건 | 게임 시작 후 5분 경과 |
| 문제 조건 | 30문제 모두 소진 |

| ID | 요구사항 |
|----|---------|
| FR-50 | 점수 조건 충족 시, 진행 중인 문제를 완료한 직후 게임을 종료한다. |
| FR-51 | 시간/문제 조건으로 종료 시, 가장 높은 점수의 플레이어가 승리한다. |
| FR-52 | 동점자가 있을 경우 "공동 우승"으로 처리한다. |
| FR-53 | 결과 화면에서 방장만 "다시 하기" 버튼을 볼 수 있다. |
| FR-54 | 다시 하기 시 점수를 초기화하고 대기실로 복귀한다. |

#### 2.1.7 플레이어 퇴장 처리

| ID | 요구사항 |
|----|---------|
| FR-60 | 게임 중 플레이어 퇴장 시 점수판에서 즉시 제거하고 전체에 알린다. |
| FR-61 | 방장이 게임 중 퇴장하면, 다음 입장 순서 플레이어가 방장 권한을 이어받는다. |
| FR-62 | 남은 플레이어가 1명 미만이 되면 게임을 즉시 종료한다. |
| FR-63 | 방에 아무도 없으면 서버에서 방 정보를 삭제한다. |

### 2.2 비기능 요구사항

| ID | 요구사항 |
|----|---------|
| NFR-01 | 정답 처리 응답은 500ms 이내여야 한다. |
| NFR-02 | 동시 최대 접속자는 방 전체 기준 10명으로 설계한다. |
| NFR-03 | 크롬, 엣지, 사파리, 파이어폭스 최신 버전을 지원한다. |
| NFR-04 | 모바일 브라우저(스마트폰)에서도 플레이 가능한 반응형 레이아웃을 제공한다. |
| NFR-05 | 서버 재시작 시 진행 중인 게임 상태가 초기화됨을 허용한다. (데이터 영속성 불필요) |

### 2.3 엣지 케이스 정의

| 상황 | 처리 방법 |
|------|-----------|
| 존재하지 않는 방 코드로 입장 | "존재하지 않는 방입니다." 오류 표시 |
| 이미 게임 중인 방에 입장 시도 | "게임이 이미 진행 중입니다." 오류 표시 |
| 같은 방 내 닉네임 중복 | "이미 사용 중인 닉네임입니다." 오류 표시 |
| 방장 혼자 남아서 게임 시작 시도 | "최소 2명 이상이 필요합니다." 버튼 비활성화 |
| 게임 중 플레이어가 1명 남음 | 게임 즉시 종료, 결과 화면 표시 |
| 네트워크 끊김 | "서버 연결이 끊어졌습니다. 새로고침 해주세요." 안내 메시지 표시 |
| 닉네임 미입력 상태로 버튼 클릭 | 입력창 테두리 강조, "닉네임을 입력해주세요." 표시 |

## 3. 화면 설계서

### 3.1 전체 화면 흐름

```
[메인 화면] ──방 만들기──────────────────────→ [대기실] (방장)
            └──방 코드 입력 후 입장──────────→ [대기실] (참여자)
            └──URL 파라미터 ?room=코드 감지──→ [메인 화면] (방코드 자동 입력)

[대기실] ──게임 시작 (방장, 2명 이상)──→ [게임 화면]

[게임 화면] ──10점 달성 / 5분 경과 / 30문제 소진──→ [결과 화면]

[결과 화면] ──다시 하기 (방장만)──→ [대기실]
```

---

### 3.2 화면 1 — 메인 화면 (닉네임 입력)

**목적:** 닉네임 설정 및 방 생성/입장 선택

**레이아웃:**

```
┌──────────────────────────────────────┐
│                                      │
│         단어 맞추기 게임             │
│                                      │
│   ┌──────────────────────────────┐   │
│   │  닉네임을 입력하세요          │   │
│   └──────────────────────────────┘   │
│                                      │
│   ┌──────────────────────────────┐   │
│   │        방 만들기             │   │  ← 버튼
│   └──────────────────────────────┘   │
│                                      │
│   ───────────── 또는 ─────────────   │
│                                      │
│   ┌────────────────────┐  ┌──────┐  │
│   │  방 코드 입력       │  │입장  │  │
│   └────────────────────┘  └──────┘  │
│                                      │
└──────────────────────────────────────┘
```

**구성 요소:**

| 요소 | 유형 | 상세 |
|------|------|------|
| 게임 타이틀 | 텍스트 | 화면 상단 중앙 |
| 닉네임 입력창 | Text Input | 최대 10자, placeholder: "닉네임을 입력하세요" |
| 방 만들기 버튼 | Button | 닉네임 미입력 시 비활성화 |
| 방 코드 입력창 | Text Input | 6자리, 영문 대문자 자동 변환 |
| 입장 버튼 | Button | 닉네임 + 방 코드 둘 다 입력 시 활성화 |
| 오류 메시지 영역 | 텍스트 | 존재하지 않는 방, 닉네임 중복 등 오류 표시 |

**동작 정의:**

| 동작 | 결과 |
|------|------|
| URL에 `?room=ABC123` 포함된 상태로 접속 | 방 코드 입력창에 `ABC123` 자동 입력 |
| 닉네임 입력 후 방 만들기 버튼 클릭 | 서버에 방 생성 요청 → 성공 시 대기실로 이동 |
| 닉네임 + 방 코드 입력 후 입장 버튼 클릭 | 서버에 입장 요청 → 성공 시 대기실로 이동, 실패 시 오류 메시지 표시 |
| Enter 키 | 현재 포커스 기준으로 제출 |

### 3.3 화면 2 — 대기실

**목적:** 게임 시작 전 참여자 모집 및 게임 시작 준비

**레이아웃:**

```
┌──────────────────────────────────────┐
│  방 코드: ABC123        [링크 복사]  │
│──────────────────────────────────────│
│                                      │
│  참여자 (3명)                        │
│  ┌────────────────────────────────┐  │
│  │  [왕관] 서덕주  (방장)         │  │
│  │         김철수                 │  │
│  │         이영희                 │  │
│  └────────────────────────────────┘  │
│                                      │
│  ┌────────────────────────────────┐  │
│  │          게임 시작             │  │  ← 방장에게만 표시
│  └────────────────────────────────┘  │
│  ※ 최소 2명 이상 필요합니다         │  ← 1명일 때만 표시
│                                      │
└──────────────────────────────────────┘
```

**구성 요소:**

| 요소 | 유형 | 상세 |
|------|------|------|
| 방 코드 표시 | 텍스트 | 상단 고정, 강조 표시 |
| 링크 복사 버튼 | Button | 클릭 시 초대 링크 클립보드 복사, 클릭 후 "복사됨!" 피드백 2초 표시 |
| 참여자 수 | 텍스트 | "참여자 (N명)" 실시간 업데이트 |
| 참여자 목록 | 리스트 | 입장 순서대로, 방장은 왕관(👑) 아이콘 표시 |
| 게임 시작 버튼 | Button | 방장에게만 표시, 1명 이하일 때 비활성화 |
| 최소 인원 안내 | 텍스트 | 1명 이하일 때만 표시 |

**실시간 이벤트:**

| 이벤트 | 화면 변화 |
|--------|-----------|
| 새 플레이어 입장 | 목록에 닉네임 추가, 참여자 수 증가 |
| 플레이어 퇴장 | 목록에서 제거, 참여자 수 감소 |
| 방장 변경 | 왕관 아이콘 이동, 새 방장에게 게임 시작 버튼 표시 |

### 3.4 화면 3 — 게임 화면

**목적:** 핵심 게임 플레이

**레이아웃:**

```
┌──────────────────────────────────────┐
│  문제 3 / 30              ⏱ 04:32   │  ← 게임 잔여시간
```

## 4. 기술 아키텍처

### 4.1 기술 스택

| 구분 | 기술 | 버전 | 선택 이유 |
|------|------|------|----------|
| 프론트엔드 | HTML5 / CSS3 / Vanilla JS | ES6+ | 설치 불필요, 모든 브라우저 지원, 초기 진입 장벽 낮음 |
| 백엔드 | Node.js | 20 LTS | 안정적인 LTS 버전 |
| 실시간 통신 | Socket.io | 4.x | WebSocket 기반 실시간 양방향 통신, 폴백 지원 |
| 웹 프레임워크 | Express.js | 4.x | 정적 파일 서빙 + 헬스체크 API |
| 단어 데이터 | JSON 파일 | - | 별도 DB 없이 파일 기반 관리 |
| 패키지 관리 | npm | - | Node.js 기본 포함 |

### 4.2 시스템 구성도

```
[브라우저 — HTML/CSS/JS]
       │
       │  WebSocket (Socket.io)
       │  HTTP (정적 파일 요청)
       ▼
[Render.com — Node.js + Express + Socket.io]
       │
       │  파일 I/O (서버 시작 시 1회)
       ▼
  [data/words.json]
```

> **왜 Render.com인가?**
>
> Cloudflare의 무료 플랜(Workers/Pages)은 정적 파일 또는 무상태(Stateless) 함수만 지원한다.  
> 이 게임은 여러 플레이어 간 실시간 상태를 공유해야 하므로, 지속적으로 실행되는 WebSocket 서버가 필요하다.  
> Cloudflare에서 이를 지원하려면 "Durable Objects" 유료 기능이 필요하다 ($5/월).  
> Render.com 무료 플랜은 Node.js + Socket.io를 추가 비용 없이 지원하므로, 이를 채택한다.

### 4.3 폴더 구조

```
word-game/
├── server.js             # Express + Socket.io 서버 진입점
├── package.json          # 의존성 및 스크립트 정의
├── .env.example          # 환경변수 예시 (실제 .env는 Git 제외)
├── render.yaml           # Render 배포 설정
├── data/
│   └── words.json        # 단어 목록 200개 (word, category)
└── public/               # 브라우저에 전달되는 정적 파일
    ├── index.html        # 단일 HTML 파일 (전체 화면 포함)
    ├── css/
    │   └── style.css     # 전체 스타일
    └── js/
        └── client.js     # 클라이언트 Socket.io 로직 및 화면 전환
```

### 4.4 서버 메모리 데이터 구조

서버는 별도 데이터베이스 없이 메모리(Map 객체)로 방 상태를 관리한다.

```
rooms: Map<string, Room>

Room = {
  code: string,                         // "ABC123"
  hostSocketId: string,                 // 방장 소켓 ID
  players: Array<{
    socketId: string,
    nickname: string,
    score: number,
    joinOrder: number                   // 방장 위임 순서 결정용
  }>,
  status: "waiting" | "playing" | "finished",
  wordQueue: Array<WordItem>,           // 섞인 출제 순서
  currentIndex: number,                 // 현재 문제 인덱스 (0-based)
  currentWord: string,                  // 현재 정답 단어
  questionTimer: NodeJS.Timeout | null, // 20초 문제 타이머
  categoryTimer: NodeJS.Timeout | null, // 15초 카테고리 힌트 타이머
  gameTimer: NodeJS.Timeout | null,     // 5분 게임 전체 타이머
  gameStartTime: number                 // Date.now()
}
```

## 5. 배포 계획

### 5.1 배포 환경

| 구분 | 서비스 | 비용 | 역할 |
|------|--------|------|------|
| 서버 | Render.com | 무료 | Node.js 서버 실행 + 정적 파일 서빙 |

> **Render.com 무료 플랜 제한사항 (알고 사용하기):**
> - 15분 이상 요청이 없으면 서버가 자동으로 절전(Sleep) 상태 전환
> - 절전 후 첫 접속 시 약 30초~1분의 콜드 스타트 발생 (첫 접속자가 기다려야 함)
> - 소규모 지인 플레이 용도로는 충분함
> - 해결책: 게임 시작 전 미리 접속해 서버를 깨워두면 됨

### 5.2 로컬 개발 환경 요구사항

| 소프트웨어 | 버전 | 확인 방법 |
|-----------|------|----------|
| Node.js | 20 LTS 이상 | `node --version` |
| npm | Node.js에 포함 | `npm --version` |
| 웹 브라우저 | 최신 버전 2개 이상 | 멀티플레이어 테스트용 |

**로컬 실행 명령:**

```bash
cd word-game
npm install
npm start
# 브라우저에서 http://localhost:3000 접속
# 두 번째 탭에서도 http://localhost:3000 접속해 멀티플레이어 테스트
```

### 5.3 Render 배포 설정

`render.yaml` 파일로 배포 설정을 코드로 관리한다.

```yaml
services:
  - type: web
    name: word-game
    runtime: node
    buildCommand: npm install
    startCommand: node server.js
    envVars:
      - key: NODE_ENV
        value: production
      - key: PORT
        value: 3000
```

**배포 순서 (Phase 6에서 진행):**
1. GitHub 저장소 생성 후 코드 업로드
2. Render.com 로그인 → "New Web Service" 생성
3. GitHub 저장소 연결 → 자동 배포 설정
4. 배포 완료 후 제공되는 URL로 접속 테스트

### Phase 2 — 닉네임 입력 및 방 시스템

**목표:** 방 만들기, 방 코드 입장, 대기실 참여자 목록이 동작하는 상태

**구현 항목:**
- `index.html`: 메인 화면 UI (닉네임 입력, 방 만들기 버튼, 방 코드 입력)
- `index.html`: 대기실 화면 UI (참여자 목록, 링크 복사, 게임 시작 버튼)
- `client.js`: 화면 전환 로직 (메인 ↔ 대기실)
- `client.js`: `create_room`, `join_room` 이벤트 송신
- `server.js`: `create_room` 처리 (방 코드 생성, rooms Map 저장)
- `server.js`: `join_room` 처리 (방 존재 확인, 게임 중 확인, 닉네임 중복 확인)
- `server.js`: `player_joined`, `player_left` 브로드캐스트
- `server.js`: 방장 퇴장 시 다음 플레이어에게 방장 위임
- `client.js`: URL `?room=` 파라미터 자동 파싱
- `style.css`: 메인 화면, 대기실 기본 스타일

**검증 항목:**
- 탭 A: 방 만들기 → 대기실 이동, 방 코드 표시 확인
- 탭 B: 방 코드 입력 후 입장 → 탭 A 대기실 참여자 목록에 탭 B 닉네임 추가 확인
- 탭 A 닫기 → 탭 B 대기실 참여자 수 감소 확인
- 탭 B가 방장이 됨 (게임 시작 버튼 표시) 확인
- 존재하지 않는 방 코드 입장 시 오류 메시지 확인
- URL `?room=ABC123` 접속 시 방 코드 자동 입력 확인
- 링크 복사 버튼 클릭 시 클립보드에 초대 링크 복사 확인

**사이드 이펙트 범위:** Phase 1 서버 구동 상태 유지 확인

### Phase 3 — 게임 핵심 로직

**목표:** 게임 시작 후 문제가 출제되고 정답을 맞출 수 있는 상태

**구현 항목:**
- `server.js`: `start_game` 처리 (방장 검증, 2명 이상 검증, 단어 목록 셔플)
- `server.js`: 힌트 생성 함수 (2글자: 앞뒤 1자 랜덤, 3자 이상: 앞뒤 2자 랜덤)
- `server.js`: `new_question` 이벤트 브로드캐스트
- `index.html` + `client.js`: 게임 화면 UI (힌트 표시, 입력창, 점수판, 정답 로그)
- `client.js`: 메인/대기실 → 게임 화면 전환
- `server.js`: `submit_answer` 처리 (정답 비교, +1점, 브로드캐스트)
- `client.js`: 정답자 발생 시 팝업 표시, 점수판 업데이트
- `server.js`: 정답 후 3초 대기 → 다음 문제 자동 전환

**검증 항목:**
- 방장이 게임 시작 버튼 클릭 → 탭 A, B 모두 게임 화면으로 전환 확인
- 힌트가 "세종ㅇㅇ" 또는 "ㅇㅇ대왕" 형식으로 표시 확인
- 2글자 단어 힌트 "ㅇ자" / "사ㅇ" 형식 확인
- 탭 A에서 정답 입력 → 탭 B에 정답자 및 정답 표시 확인
- 탭 A 점수 1점 증가, 탭 B 점수 변화 없음 확인
- 3초 후 자동으로 다음 문제 전환 확인

**사이드 이펙트 범위:** Phase 2 방 시스템 (대기실 ↔ 게임 화면 전환) 회귀 테스트

### Phase 4 — 타이머 및 게임 종료

**목표:** 20초 타이머, 15초 카테고리 힌트, 모든 종료 조건이 동작하는 상태

**구현 항목:**
- `server.js`: 문제당 20초 타이머 구현 (서버 기준)
- `server.js`: 15초 경과 시 `category_hint` 이벤트 발송
- `server.js`: 타임아웃 시 정답 공개 → 3초 후 다음 문제
- `client.js`: 타이머 바 UI (20초 게이지, 색상 변화)
- `client.js`: 카테고리 힌트 박스 표시/숨김
- `client.js`: 문제 전환 3초 카운트다운 표시
- `server.js`: 5분 전체 게임 타이머 구현
- `server.js`: 게임 잔여 시간 동기화 (`game_started` 시 서버 시간 기준)
- `client.js`: 게임 잔여 시간 MM:SS 표시
- `server.js`: 10점 달성 시 종료 조건 처리 (현재 문제 완료 후)
- `server.js`: 5분 경과 / 30문제 소진 시 즉시 종료
- `server.js`: `game_over` 이벤트 브로드캐스트 (승자, 최종 점수, 종료 이유)
- `index.html` + `client.js`: 결과 화면 UI
- `client.js`: 게임 화면 → 결과 화면 전환
- `server.js`: 다시 하기 처리 (점수 초기화, 상태 `waiting`으로 복귀)

**검증 항목:**
- 20초 타이머 게이지 카운트다운 확인
- 15초 후 카테고리 힌트 박스 표시 확인
- 20초 후 타임아웃 정답 공개 → 3초 대기 → 다음 문제 확인
- 10점 달성 후 결과 화면 이동 확인 (공동 우승 시 공동 우승 메시지)
- 5분 타이머 동작 확인 (테스트 시 5분을 30초로 단축 후 확인)
- 다시 하기 버튼 클릭 후 점수 0점으로 대기실 복귀 확인
- 방장이 아닌 플레이어에게 다시 하기 버튼 미표시 확인

**사이드 이펙트 범위:** Phase 2~3 전체 회귀 테스트 (대기실 → 게임 → 결과 → 재시작 전체 플로우)
